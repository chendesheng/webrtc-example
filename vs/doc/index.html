    包管理

    主要的注意事项：

    1. 初始化安装依赖，删除或者新加依赖，应使用 yarn
      • 初始化：yarn init
      • 安装依赖：yarn add xxxx --save 或 yarn add xxxx --dev 前者增加 dependencies 后者命令增加 devDependencies
      • 删除依赖：yarn remove xxxx
    2. 为保证项目最终代码尽量的小，理论上不增加其他第三方库（dependencies 部分）。如果需要增加依赖，需要事先讨论。

    JavaScript

    1. 枚举应该写在 constants/enums 文件夹下的各个文件里，而不建议以字面量的形式写在 constants 下单个文件里。比如，如果需要一个枚举区分按钮的种类，建议在 constants/enums 下新建一个 buttonType.js 的文件，内容如下：

      export const monitor = 0;
export const text = 1;
export const image = 2;
// ...

      这样，在其他地方引用的时候，可以写：

      import * as buttonType from './constants/enums/buttonType';

// ...
if (button.type === buttonType.text) {
  // ...
}

      而不建议在单个文件中写：

      export const buttonType = {
  monitor: 0,
  text: 1,
  image: 2,
  // ...
}

      虽然上面两者在使用的时候，写法基本都是引用 buttonType.text，但是从本质上来说是有区别的。对于前者，由于模块本身是静态的，所以编译器可以做编译分析，在编译压缩的阶段，就可以把变量名压缩掉。比如原来是 buttonType.text的变量名，压缩后就变成a了；而对于后者，由于字面量不是静态的，编译器无法做这样的优化，压缩后的代码会大不少。

    CSS

    1. 只有需要在 iframe 中引用的 CSS 才写在样式文件中，主要包括 adaptive button 和 chat window 相关的代码。对于其他部件，如 chat button，image invitation 等，直接使用 inline style，不需要另写样式文件。
    2. CSS 使用 Less 编译，直接写 CSS 语法也是可以的。
    3. CSS 文件写在 ./css/style.less 文件夹中。
    4. CSS 通过 npm run css 编译成 ./css/style.css 文件；如果需要经常修改，可以使用 npm run css:watch 命令。

    UI

    库

    使用 Preact，相应 文档。Preact 是一个类 React 库，支持绝大部分 React 语法，同时精简体积，前者只有后者的十分之一不到。理论上，React 的知识可以直接应用到 Preact 的开发中，少量不同见这里的 文档。

    使用说明

    1. 尽量只使用和 React 功能重合的部分，Preact 提供的简便方案在工程中不使用，包括：

      • 不使用作为参数传入 render 的 props 和 state，只使用 this.props 和 this.state
      • 不使用 Linked State
      • 不使用 class 属性，只是用 className

    2. 编写 UI 控件的时候，页面上需要从 preact 引用 h 和 Component 两个部分，即

      import { h, Component } from 'preact';

    扩展阅读

    1. https://preactjs.com/guide/getting-started

    单元测试

    库

    单元测试使用 jest ，相应的 文档

    命令

    1. npm run test / npm test / npm t

      运行单元测试一次

    2. npm run test:watch

      启动单元测试检测模式，会在文件修改后自动运行相应的单元测试

    规范

    1. 每一个模块文件应创建对应的单元测试文件，单元测试文件名在原文件名中加 test。如：utils.js 的单元测试是 utils.test.js

    2. 每一个单元测试应该回答以下几个问题：

      1. 在测试什么功能？
      2. 这个功能应该做什么
      3. 实际的输出是什么
      4. 应该的输出是什么
      5. 如何重现问题

      参考代码结构如下：

      describe('(API Name Here)', () => {
  beforeAll(() => {
    // 全局初始化
  });
  afterAll(() => {
    // 全局清理
  });
  beforeEach(() => {
    // 每个测试的公用初始化
  });
  afterEach(() => {
    // 每个测试的公用清理
  });
  it('should (do something)', () => {
    // 本测试初始化
    // ...

    // 测试
    const actual = '实际输出值';
    const expected = '应该的输出值';
    expect(actual).toBe(expected, '本判断的一些额外说明');
  });
});

    说明

    1. 测试环境

      测试中，process.env.NODE_ENV === 'test'

    2. 针对非生产环境代码，可以不测试

      /* istanbul ignore if */
if (process.env.NODE_ENV !== 'production') {
  // 使用上面的注释忽略这里的非生产环境代码
}

    3. 代码覆盖率需至少达到 80%，若没有达到，视作单元测试不通过
